# Evaluación y rúbrica

## Evaluación formativa

Cada módulo produce una evidencia breve:

| Módulo | Evidencia |
|---:|---|
| 1 | prompt estructurado |
| 2 | plantilla reutilizable y dos iteraciones |
| 3 | mapa de contexto |
| 4 | Project o arquitectura de espacio persistente |
| 5 | loop documentado |
| 6 | flujo multimodal verificado |
| 7 | sistema integrado y probado |

## Rúbrica del proyecto final — 100 puntos

### 1. Prompt Engineering — 20 puntos

- objetivo observable: 4;
- contexto inmediato suficiente: 4;
- tarea y formato claros: 4;
- restricciones: 4;
- criterios de aceptación: 4.

### 2. Context Engineering — 25 puntos

- separación entre información permanente y temporal: 5;
- fuentes priorizadas y vigentes: 5;
- exclusión de ruido y duplicados: 5;
- estructura de conversaciones o procesos: 5;
- regla de actualización y suficiencia: 5.

### 3. Loop Engineering — 25 puntos

- pasos coherentes: 5;
- verificador observable: 5;
- estrategia de corrección: 5;
- condición de parada y límites: 5;
- checkpoint humano: 5.

### 4. Pruebas y confiabilidad — 20 puntos

- prueba normal: 5;
- prueba con información faltante o contradictoria: 5;
- registro de errores y correcciones: 5;
- declaración de incertidumbre y riesgos: 5.

### 5. Reutilización y comunicación — 10 puntos

- otra persona puede aplicar el sistema: 5;
- presentación clara y evidencia organizada: 5.

## Niveles de desempeño

| Puntuación | Nivel | Descripción |
|---:|---|---|
| 90–100 | Sistema confiable | Es claro, reutilizable, verificable y conserva control humano. |
| 75–89 | Sistema funcional | Produce buenos resultados, pero necesita ajustes de contexto o verificación. |
| 60–74 | Interacción estructurada | Tiene un buen prompt, pero todavía no funciona como sistema. |
| Menos de 60 | Uso improvisado | Depende demasiado del primer resultado y carece de controles. |

## Preguntas de defensa

El instructor puede pedir al alumno que responda:

1. ¿Qué parte debe permanecer estable y cuál cambia en cada ejecución?
2. ¿Qué información decidiste excluir y por qué?
3. ¿Cómo detecta el sistema que el resultado está mal?
4. ¿Qué ocurre cuando falta información?
5. ¿Cuándo debe detenerse?
6. ¿Qué decisión nunca delegarías por completo?
7. ¿Qué evidencia demuestra que el sistema ahorra repetición o correcciones?

## Criterio pedagógico

No premiar prompts innecesariamente largos ni uso de términos técnicos. Premiar decisiones claras, contexto relevante, verificación real y capacidad de reutilización.
